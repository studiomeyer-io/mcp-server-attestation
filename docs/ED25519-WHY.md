# Why Ed25519, not RSA / ECDSA / libsodium

| Property | Ed25519 | Reason it matters here |
| -------- | ------- | ---------------------- |
| Key size | 32 bytes public, 32 bytes secret seed | Fits in a single hex line in a manifest, no PEM gymnastics. |
| Signature size | 64 bytes (128 hex) | Small enough to ship inside the same JSON file. |
| Speed | Fast on every modern CPU | Verification at MCP-server startup is invisible. |
| Determinism | Deterministic, no nonce | One less foot-gun than ECDSA where bad RNG leaks the key. |
| Native in Node | `node:crypto` since Node 12 | No third-party crypto dependency. Supply-chain surface stays minimal. |
| Standard | RFC 8032 | Interoperable with cosign, age, signify, OpenSSH `ssh-ed25519`. |

## Why not libsodium / NaCl wrappers

A library whose mission is to defend against marketplace-poisoning cannot afford to drag in a chain of npm dependencies that themselves can be poisoned. The fewer transitive deps we ship, the smaller the attack surface. Node's built-in Ed25519 covers all primitives we need: `generateKeyPairSync`, `sign`, `verify`, `createPublicKey`, `createPrivateKey`. We use exactly those.

## Why not RSA-2048 or ECDSA-P256

- RSA: slow verify, large keys, padding-mode footguns.
- ECDSA-P256: per-signature nonce; weak RNG silently leaks the private key (see PS3 incident, 2010).
- Ed25519: deterministic, no nonce, no padding mode, fast.

## Key handling guidance

The library does not enforce a key storage format. Hex files are the simplest path for testing. For production:

- Use a hardware token (YubiKey FIDO Ed25519, Ledger).
- Or use a KMS that exposes Ed25519 (AWS KMS, GCP KMS, HashiCorp Vault).
- Or use Sigstore's keyless flow and cross-reference Rekor entries via `mcp-attest verify --sigstore`.

The signing operation accepts any 32-byte raw seed. Adapter glue to KMS providers is left to downstream packages.
